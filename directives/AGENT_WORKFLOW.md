> **반드시** 따라야 하는 워크플로우를 정의한다.
> CLAUDE.md / AGENTS.md / GEMINI.md의 원칙을 보강한다.

> 최종 갱신: 2026-04-19

---

## 🚨 핵심 원칙: 모든 작업 후 자체점검 (Self-Check) 필수

**예외 없음.** 작은 타자 수정이든 대규모 리팩터링이든, 모든 작업 완료 후
반드시 자체점검 체크리스트를 수행하고 결과를 사용자에게 보고한다.

이유:
- 사용자가 직접 확인하기 전에 실수를 잡는다
- AI가 "했다"고 말한 것과 "실제로 한 것"의 간극을 메운다
- 바이브 코딩에서 누적 오류를 방지한다
- 사용자가 어디까지 검증됐는지 한눈에 파악 가능하다

---

## ✅ 자체점검 체크리스트 (작업 완료 직후 수행)

각 항목을 `✅` / `❌` / `N/A` 중 하나로 표시. 하나라도 `❌`면 고치고 다시 점검.

### A. 변경 범위 준수
- 요청받은 것만 수정했는가? (CLAUDE.md §2 원칙 준수, 무관한 코드 수정 없음)
- 각 파일의 변경이 사전에 승인된 diff와 일치하는가?

### B. 무결성
- 문법 오류 없음 (괄호/쉼표/JSX 닫힘, JSON 따옴표)
- 모든 import가 실제 존재하는 파일을 가리킴
- 참조된 함수/변수/i18n 키가 정의와 일치

### C. 영향도
- 기존 기능에 영향 없음 (기존 navigation 호출, 의존성 체인)
- 기존 파일의 미수정 부분이 그대로 유지됨

### D. 일관성
- 여러 파일이 함께 변경됐다면 서로 일관 (예: i18n 키가 모든 언어 파일에 존재)
- 네이밍 규칙이 기존 코드와 일치

### E. 문서
- 새 결정/설계가 `directives/` 문서에 반영됐는가?
- CLAUDE.md의 3-layer 원칙 준수?

### F. 실행 확인 (가능한 경우)
- 사용자가 즉시 실행/테스트 가능한 상태인가?
- 추가로 사용자가 해야 할 일이 명시됐는가? (빌드, 배포, 설정 등)

## 📝 보고 형식 (작업 완료 시 반드시 다음 형식으로 보고)

```
### 1. 변경 요약
| 파일 | 변경 유형 | 요약 |
|---|---|---|
| path/to/file.js | 신규/수정/삭제 | 한 줄 요약 |

### 2. 자체점검 결과
- A. 변경 범위 준수: ✅
- B. 무결성: ✅
- C. 영향도: ✅
- D. 일관성: ✅
- E. 문서: ✅ / N/A
- F. 실행 확인: ✅ / N/A

### 3. 사용자 확인 필요 사항
- [ ] (사용자가 할 일: 빌드 / 배포 / 테스트 방법)

### 4. 다음 단계 제안
- 다음에 할 일, 의사결정 필요 사항
```

## ⚙️ 예외 처리

- **점검 중 문제 발견 시**: 즉시 수정 후 재점검. 사용자에게 보고할 때 "수정함"으로 명시
- **점검이 불가능한 항목**: `N/A` 처리 + 이유 짧게 명시 (예: "앱 실행 환경 없음")
- **점검을 건너뛰어야 할 합리적 이유**: 사용자에게 미리 양해 구하고 명시적 허락 받기. 묵시적 생략 금지
